# ADR-010: HuggingFace Publication Strategy

**Date**: 2026-03-01
**Revised**: 2026-03-04
**Status**: Accepted (Revised)

## Context

The BWSK framework has produced benchmark results across 17 models with classification,
training, memory, and convergence data. To maximize reproducibility and community
engagement, we want to publish both the raw data and fully trained model checkpoints.

The original decision (2026-03-01) planned for 112 separate repos: 96 model repos
(one per training run, named `bwsk-{model-slug}-{training-mode}`) plus 16 per-architecture
results repos. During implementation it became clear that this approach created excessive
repo sprawl and made it difficult to browse results for a single model architecture.
The strategy was revised to consolidate all 6 variants for each model into a single repo
with subdirectories.

## Decision

### Repository layout under `tzervas` on HuggingFace Hub

**Dataset repos** (benchmark results as structured JSON + CSV) — unchanged from original:

- `tzervas/bwsk-benchmark-scale-sweep` — 10 transformer models, 70M to 2.7B
- `tzervas/bwsk-benchmark-architecture-diversity` — CNN, ViT, SSM, MoE results
- `tzervas/bwsk-benchmark-convergence` — 1500-step convergence with 3 seeds
- `tzervas/bwsk-benchmark-combined` — All results combined

**Model repos** (consolidated, one per architecture) — revised:

16 repos, one per trained architecture, named `tzervas/bwsk-{slug}`:

| Slug | Architecture |
|------|--------------|
| `pythia-70m` | EleutherAI/Pythia-70M |
| `pythia-160m` | EleutherAI/Pythia-160M |
| `pythia-410m` | EleutherAI/Pythia-410M |
| `pythia-1b` | EleutherAI/Pythia-1B |
| `gpt2-small` | GPT-2 Small |
| `gpt2-medium` | GPT-2 Medium |

### Repo structure for each model repo

Each `tzervas/bwsk-{slug}` repo contains:

```
bwsk-{slug}/
├── README.md                      # Comprehensive model card (auto-generated)
├── results.json                   # Aggregated results across all 6 variants
├── finetune-conventional/
│   ├── model.safetensors
│   ├── config.json
│   ├── tokenizer.json             # (language models only)
│   └── training_results.json
├── finetune-bwsk-analyzed/
│   ├── model.safetensors
│   ├── config.json
│   └── training_results.json
├── finetune-bwsk-reversible/
│   ├── model.safetensors
│   ├── config.json
│   └── training_results.json
├── scratch-conventional/
│   └── ...
├── scratch-bwsk-analyzed/
│   └── ...
└── scratch-bwsk-reversible/
    └── ...
```

The six variant subdirectories correspond to the full factorial of:
- **Experiment**: `finetune` (fine-tune from pretrained weights) vs `scratch` (train from random init)
- **Mode**: `conventional` (standard PyTorch), `bwsk-analyzed` (BWSK classification + reporting), `bwsk-reversible` (BWSK reversible backprop + activation checkpointing)

This gives 96 total training runs: 16 architectures x 3 modes x 2 experiments.

### Naming conventions

- Dataset repos: `bwsk-benchmark-{experiment-name}`
- Model repos: `bwsk-{slug}` (architecture slug only — no mode suffix)
- Variant weights: stored in subdirectory named `{experiment}-{mode}/`
- All repos tagged: `ai-s-combinator`, `bwsk`, `information-preservation`

### All repos are public

The research is open. No private repos for any benchmark data or models.

### Model cards

Auto-generated by `scripts/generate_model_cards.py` from benchmark JSON. Each README.md
includes: base model info, full S/K classification summary, per-variant training config,
memory savings table, loss curve summary, convergence statistics, and links to the
whitepaper and GitHub repo.

## What Changed from Original Decision

| Aspect | Original (2026-03-01) | Revised (2026-03-04) |
|--------|----------------------|---------------------|
| Model repo count | 112 (96 model + 16 results) | 16 (one per architecture) |
| Repo naming | `bwsk-{slug}-{mode}` | `bwsk-{slug}` |
| Variant storage | One repo per variant | Subdirectory per variant |
| Results repos | 16 separate results repos | `results.json` at repo root |
| Coverage | Representative subset (6 repos) | All 16 architectures |

## Consequences

**Easier:**
- Anyone can reproduce results by downloading a single repo per architecture
- All 6 training variants for a model are co-located — easy to compare side by side
- HuggingFace discovery via tags and model cards
- 16 repos instead of 112 is far easier to maintain and navigate
- `results.json` at repo root gives a single aggregated view without a separate repo

**Harder:**
- Subdirectory layout is less idiomatic on HuggingFace (most repos are flat)
- Upload script (`scripts/upload_to_huggingface.py`) must handle subdirectory push correctly
- Loading a specific variant requires knowing the subdirectory path

## Alternatives Considered

1. **Keep original 112-repo layout**: The per-run repos are conceptually clean but create
   excessive sprawl. Browsing 96 model repos to find all Pythia variants is cumbersome.
   Consolidated layout wins on discoverability.

2. **Weights & Biases instead of HuggingFace**: W&B is better for experiment tracking
   but HuggingFace is the standard for model/dataset sharing. Chose HF for broader reach.

3. **Private repos with access requests**: Would limit reproducibility. Open science
   is more valuable than gatekeeping.

4. **Only upload datasets, not models**: Models enable quick validation without
   re-training. The storage cost is worth the convenience.

5. **Upload to both HF and Zenodo**: Zenodo provides DOIs for citation. Could add
   later, but HuggingFace alone is sufficient for initial publication.
